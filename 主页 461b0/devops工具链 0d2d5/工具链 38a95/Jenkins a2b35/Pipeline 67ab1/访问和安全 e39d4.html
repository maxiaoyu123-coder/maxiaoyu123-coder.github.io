基础安全机制

认证, 就是用户名和密码登陆, 也叫security realm. 授权, 就是使用户拥有哪些权限. 二者结合基本满足任何的安全策略.

认证

security realm负责认证用户, 有这些选项:

  • LDAP
  • Delegate to servlet container
  • Jenkins' own user database
  • Unix user/group database

访问控制

  • Anyone can do anything
  • Legacy mode
  • Logged-in users can do anything
  • Matrix-based security
  • Project-based matrix authorization strategy

其他安全设置

  • agent 指定JNLP agent的端口是随机还是固定, 固定更加安全, 如不需要可关闭.
  • SSH(未知)

凭证

凭证是实现安全Jenkins环境的重要一步. Credentials插件提供创建和管理插件的功能.

凭证是什么呢? 凭证也称为"secret", 凡是用来对受限资源访问的值都是凭证.

包括但不限于这几种凭证:

  • 用户名和密码

    比如用来访问gitlab.

  • docker主机证书认证
  • SSH用户和私钥

    比如用于部署一台服务器.

  • 证书
  • Secret texts—tokens or other chains
  • Secret files—flat files with the credentials
  • Secret ZIP files—ZIP files with the credentials
  • Oauth

凭证可以保存在本地(默认在JENKINS_HOME), 也可保存在外部(某种用于保存和提取的应用).

关于凭证的另一个基础观点是context/上下文, 上下文将Jenkins看作是一个层级结构, 根上下文是Jenkins本身, 其他的上下文分别是job, 用户, build agent和目录.

凭证作用范围

作用范围为凭证的使用规定了用途并限制了一个范围, 比如System是提供给Jenkins实例的, 而Global是提供给Job的.

System

Jenkins实例通过这种凭证用于邮箱认证, 从节点连接等等.

Global

是凭证的默认作用范围, 通常用于作业.

❓This credential is available to the object on which the credential is associated and all objects that are children of that object. Typically you would use global-scoped credentials for things that are needed by jobs.
