环境变量

  • 分为JENKINS内置环境变量和自定义环境变量
  • JENKINS内置环境变量: 可以被看作是pipeline与Jenkins交互的媒介。

    比如，可以在pipeline中通过BUILD_NUMBER变量知道构建任务的当前构建次数。

  • 自定义的环境变量与普通变量有什么区别？
    • 在申明式脚本中, 自定义环境变量定义在environment{}; 而普通变量定义在script块中如script{ def 变量名=值 }
    • 自定义环境变量的作用域可以是pipeline范围和stage范围; 而普通变量的作用域是pipeline范围
    • 可以将pipeline正确运行所需的环境变量统一定义在environment{}中,比如: sonarqube/gitlab/nexus地址, 凭证等;
  • 查看所有内置环境变量的值：sh "printenv | sort" (jenkinsfile中运行)

    查看内置环境变量有哪些：http://主机地址:8080/env-vars.html/

  • 如环境变量未定义, 使用env前缀访问返回null, 而无前缀访问则返回异常property not found
  • 环境变量的值是string类型，即赋值时被转换成了string，需toList(), toBoolean(), toInteger(), toFloat()等转换
    • 因为在 Jenkins 文件中所有设置的值都会被当成 String
  • 引用环境变量的格式: env.BUILD_NUMBER和BUILD_NUMBER
  • 自定义环境变量名可以驼峰或者全大写
  • 在environment块中定义环境变量的方式，注意要加上引号，普通字符串单引号，其他情况双引号
    environment {
jobName = 'example'  //赋值字符串
jobName = "${JOB_NAME.split('/')[-1]}"   //赋值变量
ENV_NAME = "${env.BRANCH_NAME == "develop" ? "staging" : "production"}"  //判断赋值
LS = "${sh(script:'ls -lah', returnStdout: true).trim()}"  //step赋值
}
  • 定义环境变量的三种方式和优先级
    #申明式，如在jenkinsfile开头位置则整个pipeline可引用；如放在stage则只有该stage可引用
environment {
   var1 = value1
}

#脚本式或申明式的script{}中
env.var1 = value1

#withEnv属于step,等号两边不可空格
withEnv(["var1=value", "var2=value"]){
  sh "echo ${env.var1}"
}

    环境变量覆盖问题:

    1. withEnv可以覆盖任意环境变量
    1. stage级环境变量可以覆盖pipeline级
    1. 脚本式环境变量只能重写脚本式环境变量

  • tools{}与environment{}
    • tools{}可理解为针对工具的environment, 为工具提供环境变量PATH
    • 不用tools, 在environment中亦可实现工具PATH的添加:
      environment{
  PATH = "${SONARSCANNER_HOME}/bin:${MAVEN_HOME}/bin:${PATH}"
  SONARSCANNER_HOME = "/opt/sonar-scanner-4.2.0.1873-linux"
  MAVEN_HOME = "/opt/apache"
}
  • 如环境变量未定义, 使用env前缀访问返回null, 而无前缀访问则pipeline终止返回异常property not found
  • 在environment块中为变量赋值时, 值可以是字符串(即使数字,保存的也是字符串数字)/变量/groovy表达式:
    • groovy表达式: ENV_NAME = "${env.BRANCH_NAME == 'develop' ? 'staging' : 'production'}"
    • 变量：jobName = "${JOB_NAME.split('/')[-1]}" 记住:在使用变量时，必须用双引号
  • 在文件夹下的job，它的JOB_NAME是文件夹/JOB，如何将文件夹名字去除: JOB_NAME.split('/')[-1]

赋值: 名称 = credentials('凭证ID')

secret类型，直接引用变量名即可

用户名密码类型，这样引用：

用户：变量名_USR ; 密码：变量名_PSW

在sh中，若有凭证变量，注意，使用单引号, 因为若用双引则Groovy将进行变量替换则导致凭证泄露，console Output中也会对此进行warning

构建参数

  • 与环境变量一样, 构建参数也都是字符串类型, 无论赋数字还是布尔
  • 添加构建参数后在第一次build时,是看不到输入参数页面的, 因为参数写在pipeline脚本中,只有构建参数这部分代码被运行过,jenkins才知道.问题无法绕过.
  • 构建参数可以与环境变量同名, 所以请注意:
    • 如构建参数没有与环境变量同名: 用params.前缀或env.前缀,或无前缀,都可以访问到
    • 如同名,则env.前缀和无前缀都指向环境变量的值, 而params.前缀指向构建参数的值

    网上有这样使用的:

    
environment {
    AWS_ECR_ACCOUNT_ID="${params.AWS_ECR_ACCOUNT_ID}"
    AWS_ACCESS_KEY_ID="${params.AWS_ACCESS_KEY_ID}"
    AWS_SECRET_ACCESS_KEY="${params.AWS_SECRET_ACCESS_KEY}"
    AWS_SESSION_TOKEN="${params.AWS_SESSION_TOKEN}"
}

parameters {
    string(name: 'AWS_ECR_ACCOUNT_ID', defaultValue: '', description: 'AWS ECR account ID')
    password(name: 'AWS_ACCESS_KEY_ID', defaultValue: '', description: 'The AWS access key ID')
    password(name: 'AWS_SECRET_ACCESS_KEY', defaultValue: '', description: 'The AWS secret access key')
    password(name: 'AWS_SESSION_TOKEN', defaultValue: '', description: 'The AWS session token')
}

generic webhook trigger的参数

这里针对它所有类型的参数

  • 是环境变量, 所以env.前缀可访问, 无前缀也可访问
  • generic webhook trigger的post parameter的value filter：删除被正则匹配的部分
  • 利用git webhook参数与构建参数同名，实现：git触发时采用git参数的值; 非git触发时采用构建参数的值

    因为git webhook参数是环境变量，当它定义时，env.前缀和无env.前缀都指向git webhook参数的值，而不会指向同名的构建参数

可能会用到自定义环境变量

  • JAVA_TOOL_OPTIONS
    • 名字里虽有“TOOL”，但是其角色是“JAVA_OPTIONS”。类似java命令行中这样的选项
    • 有时候需要调整jvm配置，但没有办法通过配置文件或者java命令行选项来做到这一点
      • 比如，Impala: Real-time Query for Hadoop，一个守护进程，管理员没有发现配置方法。此时要给Impala启动的JVM传递参数，只有使用JAVA_TOOL_OPTIONS环境变量了
      • 比如，对Jenkins容器agent的maven目录，传入JAVA_TOOL_OPTIONS=“-Duser.home=/var/maven”
