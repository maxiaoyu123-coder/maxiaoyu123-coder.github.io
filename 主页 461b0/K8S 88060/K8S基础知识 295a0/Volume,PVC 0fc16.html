  • Local Volume

为什么需要Volume？有状态的应用会保存数据文件，我们必须搭配Volume来保存它们，防止Pod重启后数据丢失.

容器Volume是宿主机上的目录，容器向Volume写数据就是向宿主机的目录写数据，类似软连接。

Volume的属性

  • 模式，分为filesystem和block

    filesystem模式卷是挂载到Pod的目录上；block模式的卷是为Pod提供块设备，Pod中的应用将直接访问块设备，不经过任何文件系统层。

  • 访问模式：RWO,RWX,ROX

Persistent Volume中的capacity.storage 字段是让PVC能够匹配到合适容量大小的Volume，一个类似标签的作用，它不限制Volume的最大使用空间，实际使用量由Volume所在存储设备的可用量决定。如果需要限制Volume使用量，可以使用分区或利用文件系统的限制功能来实现。

A hostPath volume mounts a file or directory from the host node's filesystem into your Pod. So, if you have a multi-node cluster, the pod is restarted for some reasons and assigned to another node, the new node won't have the old data on the same path. That's why we have seen, that hostPath volumes work well only on single-node clusters. Here, the Kubernetes local persistent volumes help us to overcome the restriction and we can work in a multi-node environment with no problems. It remembers which node was used for provisioning the volume, thus making sure that a restarting POD always will find the data storage in the state it had left it before the reboot. Once a node has died, the data of both hostpath and local persitent volumes of that node are lost.

Local Persistent Volume

  • 官方参考：https://kubernetes.io/blog/2019/04/04/kubernetes-1.14-local-persistent-volumes-ga/#how-to-use-a-local-persistent-volume
  • 是一种使用宿主机本地存储的Volume，可以是块设备/分区/目录。虽然HostPath也使用宿主机本地存储，但不同的是一旦与Local Persistent Volume绑定，Pod将始终调度在这个Node上，保证了数据的持久化，这是HostPath无法做到的。
  • 相比远程文件系统类型的Volume，Local Persistent Volume拥有更高的IO。但是一旦Node故障，Pod不会被调度到其他Node，将处于Pending状态。所以Local Persistent Volume适用于有状态的应用, 且该应用能承受节点失效,节点数据服务不可访问,节点数据丢失。
  • 适用的应用，这些有状态应用为集群中的其他应用提供了关键的低延迟的基础服务,所以应当具有更高的优先级.
    • 分布式存储:Cassandra,比如分布式文件系统:CEPH和GlusterFS
    • 数据集的缓存

    1. 存储类

      启动灵活调度：在创建PVC前, 创local存储类, 参考：https://kubernetes.io/docs/concepts/storage/storage-classes/#local

      #存储类
apiVersion: storage.k8s.io/v1
kind: StorageClass
metadata:
  name: local-storage
provisioner: kubernetes.io/no-provisioner   #关键属性，哪个voluem plugin创建PV
volumeBindingMode: WaitForFirstConsumer  #另一选项是Immediate:一旦pvc出现就立即用pv与其绑定，WaitForFirstConsumer:延迟绑定，直到拥有pvc的pod调度成功
    1. PV
      #PV的申明
apiVersion: v1
kind: PersistentVolume
metadata:
  name: elastic-system-pv #调整
spec:
  capacity:
    storage: 2Gi #调整
  accessModes:
  - ReadWriteOnce
  persistentVolumeReclaimPolicy: Retain
  storageClassName: local-storage
  local:
    path: /mnt/data #调整
  nodeAffinity: #调整             
   required:
     nodeSelectorTerms:
     - matchExpressions:
       - key: kubernetes.io/hostname
         operator: In
         values:
         - master01

Storage Class

一个大规模的 Kubernetes 集群里很可能有成千上万个 PVC，这就意味着运维人员必须得事先创建出成千上万个 PV。更麻烦的是，随着新的 PVC 不断被提交，运维人员就不得不继续添加新的、能满足条件的 PV，否则新的 Pod 就会因为 PVC 绑定不到 PV 而失败。在实际操作中，这几乎没办法靠人工做到。Kubernetes 为我们提供了一套可以自动创建 PV 的机制，即：Dynamic Provisioning。Dynamic Provisioning 机制工作的核心，在于一个名叫 StorageClass 的 API 对象。

StorageClass 对象会定义如下两个部分内容: 第一，PV 的属性。比如，存储类型、Volume 的大小等等;第二创建这种 PV 需要用到的存储插件。比如，Ceph 等等。 有了这样两个信息之后，Kubernetes 就能够根据用户提交的 PVC，找到一个对应的 StorageClass 了。然后，Kubernetes 就会调用该 StorageClass 声明的存储插件，创建出需要的 PV。
