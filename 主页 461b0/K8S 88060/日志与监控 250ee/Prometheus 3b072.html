5.2. 二进制部署或容器部署

5.2.1. 二进制安装

验证hash

sha256sum /tmp/alertmanager-0.21.0.linux-amd64.tar.gz

解压到安装目录

tar xf /tmp/alertmanager-0.21.0.linux-amd64.tar.gz -C /opt/
ln -s /opt/alertmanager-0.21.0.linux-amd64 /opt/alertmanager

开机自启

cat <<EOF >/usr/lib/systemd/system/alertmanager.service
[Unit]
Description=Prometheus Alertmanager Service daemon
After=network.target

[Service]
Type=simple
User=root
Group=root
ExecStart=/opt/alertmanager/alertmanager \
 --config.file=/opt/alertmanager/alertmanager.yml \
 --storage.path=/opt/alertmanager/data/ \
 --data.retention=120h \
 --web.external-url=http://192.168.186.7:9093 \
 --web.listen-address=0.0.0.0:9093
Restart=on-failure

[Install]
WantedBy=multi-user.target
EOF

5.3. 告警配置

5.3.1. Prometheus关联

编辑Prometheus的主配置文件prometheus.yml

加入关联Alertmanager组件的访问地 址，示例如下:

alerting:
  alertmanagers:
  - static_configs:
    - targets:
      - 127.0.0.1:9093   #alertmanager与Prometheus在一个主机上

在prometheus.yml文件中创建一个 Prometheus job来监控Alertmanager组件示例地址，内容如下：

scrape_configs:
  - job_name: 'alertmanager'
      static_configs:
      - targets: ['127.0.0.1:9093']   #alertmanager与Prometheus在一个主机上

在prometheus.yml文件中创建rule_files, 指定在哪个目录加载rule文件.

rule_files:
  - "/opt/prometheus/rules/*_rules.yml"

创建rule目录文件,

mkdir /opt/prometheus/rules/
vi

以创建一个node UP告警为例

创建一个rule文件.

cat <<EOF >/opt/prometheus/rules/node_rules.yml
groups:
- name: UP
  rules:
  - alert: node
	    expr: up{job="node"} == 0
    for: 3m
    labels:
      severity: critical
    annotations:
      description: Node has been down for more than 5 minutes.
      summary: Node down
EOF

检查一下语法

#promtools与prometheus在同一级目录下
./promtool check rules rules/up_rules.yml

热重启prometheus, 然后访问http://地址/rules查看rule是否生效.

关闭node-01的node_exporter, 然后观察一下http://地址:9090/alerts, 出现了一条状态为pending的告警.

一个告警信息在生命周期内可能会有三种状态

·Interval，没有满足触发条件，告警未激活状态。 ·Pending，已满足触发条件，但未满足告警持续时间的状态，即未 满足告警中for子句中指定的持续时间。 ·Firing，已满足触发条件且已经超过for子句中指定的持续时间时的 状态。带有for子句的告警将首先转换为Pending状态，然后转换为Firing状 态，至少需要两个计算周期告警才被触发。从Pending状态到Firing状态 的转换，确保了告警的有效性。而没有for子句的告警会自动从Inactive 状态转换为Firing状态，只需要一个计算周期即可被触发。

在一系列的运行过程中，Prometheus还将为Pending或Firing状态的 每个告警创建一个ALERTS指标名称，格式如下,当告警处于激活状态（Pending或Firing）时，样本值为1，其他状 态时样本值为0。

当告警信息被发送到指定的Alertmanager组件后，我们使用浏览器 访问Alertmanager Web示例地址http://192.168.186.7:9093/#/alerts，也可 以查看接收到的告警信息.

Prometheus从收集监控目标信息开始，到 触发一条告警的过程

1）定义规则。 在Prometheus配置文件中，配置scrape_interval:15s（默认值是 1min）为收集监控目标信息的采集周期，并且配置对应的告警规则。 scrape_interval可以是全局设置，也可以是单metric定义。 2）周期计算。 对相应的表达式进行计算时，在Prometheus配置文件中配置 evaluation_interval:15s（默认值是1min）为告警规则计算周期， evaluation_interval只是全局计算周期值。 3）告警状态转换。 ·当首次发现告警规则条件成立，即表达式为真，并且没有满足告 警规则中for子句中指定的持续时间时，告警状态切换为PENDING。 ·若下一个计算周期中，表达式仍然为真，并且满足告警规则中for 子句中指定的持续时间时，告警状态转换成FIRING，即为active，告警 被Prometheus发送到Alertmanager组件。若下下个计算周期，表达式还是为真，并且满足告警规则中for子 句中指定的持续时间时，持续发送告警到Alertmanager组件。 ·直到某个计算周期，表达式为假，告警状态会变为inactive，并且 会有一个resolve被发送给Altermanger，用于说明此告警已解决。

容器指标采集

cAdvisor介绍

cAdvisor可提供数据的可视化界面，并且可监控容器所在宿主 机中的资源使用情况，只监控单一主机。只需在宿主机上部署cAdvisor容器，用户就可通过Web界面或REST 服务访问当前节点和容器的详尽性能数据（CPU、内存、网络、磁盘、 文件系统等）。

在容器集群的环境中，通过Docker stats命令对一台台机器收集容器 的度量指标显然是不现实的。在Kubenetes（有时简称k8s）设计之初就 考虑到了这一点，因此它将Google开源的一个容器指标收集工具cAdvisor集成到了Kubelet中。在集群中，Kubelet会驻留在集群的每一个 节点上，因此cAdvisor也将随之运行在所有节点上。cAdvisor使用Go语言开发，利用Linux的cgroups获取容器的资源使用信息.

cAdvisor可以对节点机器上的资源及容器进行实时监控和性能数据采集，包括CPU使用情况、内存使用情况、网络吞吐量及文件系统使用 情况, 还提供基础查询界面和 http接口, 同时它消耗的资源也比较少. cAdvisor内置了对Prometheus的支持.

cAdvisor也有一定的局限性，即只能监控一个Docker主机。对于多节点的场景，就需要在所有的主机上都各安装一个 cAdvisor。另外，在图表中的数据仅仅显示时长为2分钟，并没有查看长期数据趋势的方法。它也没有生成告警的机制，如果资源使用率较高则 可能存在风险。采用cAdvisor与Prometheus结合的解决方案是一个非常 不错的选择.

安装

cAdvisor安装非常简单.

二进制方式

mkdir /opt/cadvisor-v0.37.5
curl -L https://github.com/google/cadvisor/releases/download/v0.37.5/cadvisor -o /opt/cadvisor-v0.37.5/cadvisor && \
cd /opt/cadvisor-v0.37.5/ && ./cadvisor -port=8080 >>/var/log/cadvisor.log

容器方式

docker pull google/cadvisor:latest

部署完成后, http://地址:8080/metrics就能查看到cAdvisor输出的容器指标数据了, 接下来再让Prometheus取走数据.

cAdvisor+Prometheus+Grafana组合

通过Node exporter抓取物理主机运行的状态，容器则通过 cAdvisor对Docker引擎抓取容器的运行状态，然后汇总到Prometheus， 再通过AlertManager实现告警，形成完整的解决方案

helm方式部署Prometheus

传统方式部署步骤相对复杂，使用helm方式部署Prometheus更加简单.

部署前需要安装好helm软件和Persistent Volume.

在K8S的管理节点(能够用admin账号访问的)上添加两个repo.

helm repo add prometheus-community https://prometheus-community.github.io/helm-charts
helm repo add kube-state-metrics https://kubernetes.github.io/kube-state-metrics
helm repo update

创建一个名为monitoring的namespace

kubectl create namespace monitoring

部署Prometheus

helm install kube-prometheus prometheus-community/prometheus -n monitoring
