部署API-Server

可执行文件

将可执行文件从压缩包目录移动到/opt/kubernetes/bin/目录下。

创建service文件

#service文件
cat>/usr/lib/systemd/system/kube-apiserver.service <<'EOF'
[Unit]
Description=Kubernetes API Service
Documentation=https://github.com/GoogleCloudPlatform/kubernetes
After=network.target
After=etcd.service

[Service]
EnvironmentFile=-/opt/kubernetes/cfg/kube-apiserver.conf
ExecStart=/opt/kubernetes/bin/kube-apiserver $KUBE_API_ARGS
Restart=on-failure
Type=notify
LimitNOFILE=65536

[Install]
WantedBy=multi-user.target
EOF

#配置文件
cat >/opt/kubernetes/cfg/kube-apiserver.conf <<'EOF'
KUBE_API_ARGS="--logtostderr=true \
--v=2 \
--enable-bootstrap-token-auth=true \
--token-auth-file=/opt/kubernetes/cfg/token.csv \
--kubelet-client-certificate=/opt/kubernetes/pki/kubeapi-server.pem \
--kubelet-client-key=/opt/kubernetes/pki/kubeapi-server-key.pem \
--etcd-servers=https://etcd-1:2379,https://etcd-2:2379,https://etcd-3:2379 \
--etcd-cafile=/opt/etcd/pki/etcd-ca.pem \
--etcd-certfile=/opt/etcd/pki/etcd-client-kubeapi.pem \
--etcd-keyfile=/opt/etcd/pki/etcd-client-kubeapi-key.pem \
--tls-cert-file=/opt/kubernetes/pki/kubeapi-server.pem \
--tls-private-key-file=/opt/kubernetes/pki/kubeapi-server-key.pem \
--bind-address=192.168.31.63 \
--secure-port=6443 \
--advertise-address=192.168.31.63 \
--client-ca-file=/opt/kubernetes/pki/kubeapi-ca.pem \
--enable-bootstrap-token-auth=true \
--authorization-mode=RBAC,Node \
--service-account-key-file=/opt/kubernetes/pki/kubeapi-ca-key.pem \
--anonymous-auth=false \
--allow-privileged=true \
--service-cluster-ip-range=10.0.0.0/24"
  • --client-ca-file=/opt/kubernetes/pki/ca.pem 指定认证, CA证书用于验证提交给 API server 的客户端证书.如果客户端证书已提交并验证，则使用 subject 的 Common Name（CN）作为请求的用户名。从 Kubernetes 1.4开始，客户端证书还可以使用证书的 organization 字段来指示用户的组成员身份。要为用户包含多个组成员身份，请在证书中包含多个 organization 字段。
  • --advertise-address
  • --allow-privileged=true 如果不设置, 在部署网络插件时daemonset将会报错.
  • --service-node-port-range=30000-32767
  • --enable-admission-plugins=NamespaceLifecycle,LimitRanger,ServiceAccount,ResourceQuota,NodeRestriction
  • --audit-log-maxage=30 --audit-log-maxbackup=3 -audit-log-maxsize=100 --audit-log-path=/opt/kubernetes/logs/k8s-audit.log

关于kublet-bootstrap功能的介绍

该功能让kubelet可以自动向apiserver申请证书, apiserver可代表CA向kubelet签发证书，不需要管理者手动签发证书了. 在向apiserver申请证书的过程中，kubelet以token方式通过认证，以一个低级用户身份向apiserver发出CSR证书签发请求。

创建token.csv文件

system:node-bootstrapper角色是预定义的，只能用来作kublet-bootstrap的申请证书，将其赋予kubelet-bootstrap用户。

kubectl create clusterrolebinding kubelet-bootstrap \
--clusterrole=system:node-bootstrapper \
--user=kubelet-bootstrap

kube-apiserver会引用token.csv文件，如此kubelet可通过token方式通过认证，并以kubelet-bootstrap用户身份进行证书申请。

#token号码是我随机生成的一份, 直接使用即可.

cat <<EOF >/opt/kubernetes/cfg/token.csv
bc47661b5bad82e72f0cd646be261d5d,kubelet-bootstrap,10001,"system:kubelet-bootstrap"
EOF
#kublet向apiserver出示这个token,认证通过后以kubelet-bootstrap用户为自己颁发证书.
#这里的token必须与node节点bootstrap.kuconfig配置里的一致.

启动

systemctl enable --now kube-apiserver

keepalived

#安装
yum install keepalived

#LB01上添加
cat /etc/keepalived/keepalived.conf <<'EOF'
global_defs {
    router_id lb01			#标识符，一般设置为主机名
    script_user root
    enable_script_security
}
vrrp_script check_nginx {
    script  "/etc/keepalived/script/check_nginx.sh"
    interval  5					#脚本执行间隔
    weight  -30					#脚本执行错误优先级减30
}
vrrp_instance VI_1 {
    state MASTER					#状态可以是MASTER和BACKUP两种
    nopreempt
    interface eth0					#实例绑定的网卡
    unicast_src_ip 192.168.31.61
    unicast_peer {
    	192.168.31.62
    }
    virtual_router_id 51				#两台服务器id值必须相同
    priority 100					#优先级，优先级高的获得VIP
    advert_int 1					#检查间隔，即VRRP包发送的间隔
    authentication {					#认证
        auth_type PASS
        auth_pass 1111
    }
    virtual_ipaddress {
        192.168.31.60		  #VIP
    }
    track_script {
        check_nginx
    }
}
EOF

#LB02上添加
cat /etc/keepalived/keepalived.conf <<'EOF'
global_defs {
    router_id  lb02           #标识符，一般设置为主机名
    script_user root
    enable_script_security
}
vrrp_script check_nginx {
    script  "/etc/keepalived/script/check_nginx.sh"
    interval  5                 #脚本执行间隔
    weight  -40                 #脚本执行错误优先级减20
}
vrrp_instance VI_1 {
    state BACKUP                    #状态可以是MASTER和BACKUP两种
    interface eth0                  #实例绑定的网卡
    unicast_src_ip 192.168.31.62
    unicast_peer {
        192.168.31.61
    }
    virtual_router_id 51                #两台服务器id值必须相同
    priority 90                 #优先级，优先级高的获得VIP
    advert_int 1                    #检查间隔，即VRRP包发送的间隔
    authentication {                    #认证
        auth_type PASS
        auth_pass 1111
    }
    virtual_ipaddress {
        192.168.31.60           #VIP
    }
    track_script {
        check_nginx
    }
}
EOF

systemctl enable --now keepalived

部署Controller-manager

可执行文件

将可执行文件从压缩包目录移动到/opt/kubernetes/bin/目录下。

创建kubeconfig文件

--server=https://192.168.31.60:6443 指向NGINX上的代理地址

touch /opt/kubernetes/cfg/controller-manager.conf
KUBECONFIG=/opt/kubernetes/cfg/controller-manager.conf kubectl config set-cluster default-cluster --server=https://192.168.31.60:6443 --certificate-authority /opt/kubernetes/pki/kubeapi-ca.pem --embed-certs
KUBECONFIG=/opt/kubernetes/cfg/controller-manager.conf kubectl config set-credentials default-controller-manager --client-key /opt/kubernetes/pki/kube-apiserver-client-kcm-key.pem --client-certificate /opt/kubernetes/pki/kube-apiserver-client-kcm.pem --embed-certs
KUBECONFIG=/opt/kubernetes/cfg/controller-manager.conf kubectl config set-context default-system --cluster default-cluster --user default-controller-manager
KUBECONFIG=/opt/kubernetes/cfg/controller-manager.conf kubectl config use-context default-system

添加Service文件

#添加service文件
cat>/usr/lib/systemd/system/kube-controller-manager.service <<'EOF'
[Unit]
Description=Kubernetes Controller Manager
Documentation=https://github.com/GoogleCloudPlatform/kubernetes

[Service]
EnvironmentFile=/opt/kubernetes/cfg/kube-controller-manager.conf
ExecStart=/usr/local/bin/kube-controller-manager $KUBE_CONTROLLER_MANAGER_ARGS
Restart=on-failure
LimitNOFILE=65536

[Install]
WantedBy=multi-user.target
EOF

#添加service需要引用的包含配置选项的文件
cat >/opt/kubernetes/cfg/kube-controller-manager.conf <<'EOF'
KUBE_CONTROLLER_MANAGER_ARGS="--logtostderr=true \
--v=2 \
--log-dir=/opt/kubernetes/logs \
--leader-elect=true \
--master=127.0.0.1:8080 \
--address=127.0.0.1 \
--allocate-node-cidrs=true \
--cluster-cidr=10.244.0.0/16 \
--service-cluster-ip-range=10.0.0.0/24 \
--cluster-signing-cert-file=/opt/kubernetes/pki/kubeapi-ca.pem \
--cluster-signing-key-file=/opt/kubernetes/pki/kubeapi-ca-key.pem \
--root-ca-file=/opt/kubernetes/pki/kubeapi-ca.pem \
--service-account-private-key-file=/opt/kubernetes/pki/kubeapi-ca-key.pem \
--kubeconfig=/opt/kubernetes/cfg/controller-manager.conf \
--experimental-cluster-signing-duration=87600h0m0s"
EOF
  • --kubeconfig

    日志中提示了No authentication-kubeconfig provided in order to lookup requestheader-client-ca-file in configmap/extension-apiserver-authentication in * kube-system, so request-header client certificate authentication won't work. No authentication-kubeconfig provided in order to lookup client-ca-file in configmap/extension-apiserver-authentication in kube-system, so client certificate authentication won't work. No authentication-kubeconfig provided in order to lookup client-ca-file in configmap/extension-apiserver-authentication in kube-system, so client certificate authentication won't work.

  • --root-ca-file 用来对 kube-apiserver 证书进行校验，指定该参数后，才会在Pod 容器的 ServiceAccount 中放置该 CA 证书文件；\
  • --cluster-signing-* 指定的证书和私钥文件用来签名为 TLS BootStrap 创建的证书和私钥；
  • --service-cluster-ip-range 指定集群Service的CIDR范围，该网络在各 Node 间必须路由不可达，必须和 kube-apiserver 中的参数一致；
  • --bind-address 值必须为 127.0.0.1，kube-apiserver 期望 scheduler 和 controller-manager 在同一台机器；
  • --service-account-private-key-file
  • --experimental-cluster-signing-duration
  • --master=127.0.0.1:8080

启动

systemctl enable --now kube-controller-manager

部署Scheduler

创建kubeconfig文件

scheduler服务访问apiserver时所需的kubeconfig文件, 与kubectl访问时的情况类似.

--server=https://192.168.31.60:6443 指向NGINX上的代理地址。

touch /opt/kubernetes/cfg/scheduler.conf
KUBECONFIG=/opt/kubernetes/cfg/scheduler.conf kubectl config set-cluster default-cluster --server=https://192.168.31.60:6443 --certificate-authority /opt/kubernetes/pki/kubeapi-ca.pem --embed-certs
KUBECONFIG=/opt/kubernetes/cfg/scheduler.conf kubectl config set-credentials default-scheduler --client-key /opt/kubernetes/pki/kube-apiserver-client-scheduler-key.pem --client-certificate /opt/kubernetes/pki/kube-apiserver-client-scheduler.pem --embed-certs
KUBECONFIG=/opt/kubernetes/cfg/scheduler.conf kubectl config set-context default-system --cluster default-cluster --user default-scheduler
KUBECONFIG=/opt/kubernetes/cfg/scheduler.conf kubectl config use-context default-system

添加service文件

添加service文件
cat >/usr/lib/systemd/system/kube-scheduler.service <<'EOF'
[Unit]
Description=Kubernetes Scheduler Plugin
Documentation=https://github.com/GoogleCloudPlatform/kubernetes

[Service]
EnvironmentFile=/opt/kubernetes/cfg/kube-scheduler.conf
ExecStart=/opt/kubernetes/bin/kube-scheduler $KUBE_SCHEDULER_ARGS
Restart=on-failure
LimitNOFILE=65536

[Install]
WantedBy=multi-user.target
EOF

#添加service需要引用的包含配置选项的文件
cat >/opt/kubernetes/cfg/kube-scheduler.conf <<'EOF'
KUBE_SCHEDULER_ARGS="--logtostderr=true \
--v=2 \
--log-dir=/opt/kubernetes/logs \
--leader-elect=true \
--kubeconfig=/opt/kubernetes/cfg/scheduler.conf \
--master=127.0.0.1:8080 \
--address=127.0.0.1"
EOF

启动

systemctl enable --now kube-scheduler
