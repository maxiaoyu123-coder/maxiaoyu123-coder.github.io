TLS证书的类型

peer

要求服务端和客户端在证明彼此的身份之后才能建立TLS连接。

etcd成员之间通信采用了peer类型的证书, 实现彼此的身份验证和通信加密.

server

服务端使用server证书向客户端证明自己的身份, 并且实现服务端与客户端之间的数据的加密. 客户端为了能够验证服务端的身份, 需要本地保存一份服务端证书的颁发机构CA们的证书, 即certificate authorities list.

The server certificate should have its hostname as CN (Common Name). For example: if you want to access the server https://www.mycompany.com the CN should be www.mycompany.com.

client

客户端使用client证书向客户端证明自己的身份, 但是不实现客户端与服务端的数据的加密.

Etcd CA与Kubernetes CA

Etcd节点通信，包括服务端与客户端/服务端与服务端之间的，所用的证书由etcd专用CA颁发。而k8s apiserver与其他组件之间通信所用的证书由k8s专用CA颁发。这样作的目的是etcd服务器只允许etcd服务器和客户端的访问, 试想假如kubelet证书与etcd证书都由一个CA颁发, kubelet可向etcd证明自己身份, 而etcd也可向kubelet证明身份, 这样kubelet与etcd就能进行通信了.

etcd CA

etcd CA需要颁发如下证书:

  • 一份证书用于etcd节点间互相证明身份, 以及向客户端证明身份.. 证书类型是peer.
  • 一份证书用于为apiserver向etcd证明身份. apiserver作为client用的证书是client类型.

准备步骤：创建目录

mkdir -p /opt/etcd/{bin,cfg,logs,pki}

步骤一: etcd CA证书和签发所用的配置文件

etcd CA的证书有效期是10年.

names字段是什么意思?

cat >/opt/etcd/pki/etcd-ca-csr.json <<EOF
{
    "CN": "Kubernetes by Parts: etcd CA",
    "key": {
        "algo": "rsa",
        "size": 2048
    },
    "ca": {
        "expiry": "87600h"
    },
    "names": [
        {
            "L": "Kubernetes by Parts",
            "O": "Clusterise",
            "ST": "etcd"
        }
    ]
}
EOF

cd /opt/etcd/pki/
cfssl gencert -initca /opt/etcd/pki/etcd-ca-csr.json | cfssljson -bare etcd-ca -

etcd CA签发证书所用的配置文件. 三个profile分别用于签发server类型, client类型和peer类型的证书, 所有证书都是10年.

cat > /opt/etcd/pki/ca-config.json <<EOF
{
  "signing": {
      "default": {
          "expiry": "87600h"
      },
      "profiles": {
          "server": {
              "expiry": "87600h",
              "usages": [
                  "signing",
                  "key encipherment",
                  "server auth"
              ]
          },
          "client": {
              "expiry": "87600h",
              "usages": [
                  "signing",
                  "key encipherment",
                  "client auth"
              ]
          },
          "peer": {
              "expiry": "87600h",
              "usages": [
                  "signing",
                  "key encipherment",
                  "server auth",
                  "client auth"
              ]
          }
      }
  }
}
EOF
#509证书中的信息:
#CN: CommonName指的是CA的名称.
#OU: OrganizationalUnit 指的是部门
#O: Organization 指的是公司
#L: Locality指城市
#S: StateOrProvinceName
#C: CountryName
#比如CN=Sample Cert, OU=R&D, O=Company Ltd., L=Dublin 4, S=Dublin, C=IE
#client auth表示client可以用该 CA 对server提供的证书进行验证；
#server auth表示server可以用该CA对client提供的证书进行验证

步骤二: 签发一份etcd的证书

此证书用于etcd节点间互相证明身份, 以及向客户端证明身份。-profile=peer 表示在作客户端或服务端时，此证书皆可用来证明自己的身份，一证两用。

为了简化部署, 所有etcd节点都用一份证书, 在hosts字段加入主机名(地址也可)即SANs.

补充知识：CN(CommonName)字段代表证书出示者的域名,早先在验证证书时验证者会核查CN与出示者的域名(比如浏览器将url中的域名与CN比较)是否精确匹配, 相当于机房出入证上你的名字, 但是CN字段里只能写一个名字, 无法写一个名字列表, 相当于出入证上只能写一个人的名字只能被一个人用, 一组人的话需每人签发一份, 不能共用. 之后为了解决这个限制, 证书中增加了一个字段, 叫SAN(Subject Alternate Name), 是个包含地址或域名的列表, 如果证书有hosts字段那么验证者忽略CN字段而只核对hosts字段中的地址(域名). 这样一组主机只签发一个共用的证书即可, 证书管理上也方便了 . 在csr文件中hosts字段就是SAN字段. 

cat << EOF > /opt/etcd/pki/etcd-peer-server-csr.json
{
  "CN": "etcd",
  "hosts": [
    "etcd-1",
    "etcd-2",
    "etcd-3",
    "127.0.0.1"
  ],
  "key": {
      "algo": "rsa",
      "size": 2048
  },
  "names": [
      {
          "L": "Kubernetes by Parts",
          "O": "Clusterise",
          "ST": "etcd",
          "OU": "etcd peering and server"
      }
  ]
}
EOF

cd /opt/etcd/pki/
cfssl gencert -ca=/opt/etcd/pki/etcd-ca.pem -ca-key=/opt/etcd/pki/etcd-ca-key.pem -config=/opt/etcd/pki/ca-config.json -profile=peer /opt/etcd/pki/etcd-peer-server-csr.json | cfssljson -bare etcd-peer-server-csr

etcdctl测试一下

/opt/etcd/bin/etcdctl \
--ca-file=/opt/etcd/pki/etcd-ca.pem \
--cert-file=/opt/etcd/pki/etcd-peer-server-csr.pem \
--key-file=/opt/etcd/pki/etcd-peer-server-csr-key.pem \
--endpoints="https://etcd-1:2379" \
cluster-health

将这份证书复制到etcd-2和etcd-3.


scp -p /opt/etcd/pki/{etcd-ca.pem,etcd-peer-server-csr.pem,etcd-peer-server-csr-key.pem} etcd-2:/opt/etcd/pki/
scp -p /opt/etcd/pki/{etcd-ca.pem,etcd-peer-server-csr.pem,etcd-peer-server-csr-key.pem} etcd-3:/opt/etcd/pki/

签发controller-manager的证书

用于访问apiserver时提供客户端证书.

controller-mgr以system:kube-controller-manager用户(写在CN),以system:kube-controller-manager用户组(写在OU), 访问apiserver. 换而言之, apiserver根据客户端证书中的CN和OU两个字段获知客户端的用户以及所属用户组, 依此判断客户端拥有何种权限.

cat << EOF > /opt/kubernetes/pki/kube-apiserver-client-kcm-csr.json
{
    "CN": "system:kube-controller-manager",
    "key": {
        "algo": "rsa",
        "size": 2048
    },
    "names": [
        {
            "L": "Kubernetes by Parts",
            "O": "Clusterise",
            "ST": "kube-apiserver",
            "OU": "kube-controller-manager"
        }
    ]
}
EOF

cfssl gencert -ca=kubeapi-ca.pem -ca-key=kubeapi-ca-key.pem -config=ca-config.json -profile=client /opt/kubernetes/pki/kube-apiserver-client-kcm-csr.json | cfssljson -bare kube-apiserver-client-kcm

签发admin证书

kubelet 访问 kube-apiserver 时 ，使用该证书通过认证，并因为"O": "system:masters"获得system:masters权限，拥有访问所有 API 的权限。

cat << EOF > /opt/kubernetes/pki/kube-apiserver-client-admin-csr.json
{
    "CN": "kubernetes-admin",
    "key": {
        "algo": "rsa",
        "size": 2048
    },
    "names": [
        {
            "L": "Kubernetes by Parts",
            "O": "system:masters",
            "ST": "kube-apiserver",
            "OU": "admin"
        }
    ]
}
EOF

cd /opt/kubernetes/pki && \
cfssl gencert -ca=/opt/kubernetes/pki/kubeapi-ca.pem -ca-key=/opt/kubernetes/pki/kubeapi-ca-key.pem -config=/opt/kubernetes/pki/ca-config.json -profile=client /opt/kubernetes/pki/kube-apiserver-client-admin-csr.json | cfssljson -bare kube-apiserver-client-admin
