安装Docker

  • 步骤
    #安装docker
yum install -y yum-utils
yum-config-manager --add-repo https://mirrors.aliyun.com/docker-ce/linux/centos/docker-ce.repo
yum remove -y docker docker-common docker-selinux docker-engine
yum list docker-ce --showduplicates 
yum install -y docker-ce-18.09.9
mkdir /etc/docker
cat>/etc/docker/daemon.json<<EOF
{
        "registry-mirrors": ["https://registry.docker-cn.com"],
        "exec-opts":["native.cgroupdriver=systemd"]
}
EOF
systemctl daemon-reload
systemctl start docker && systemctl enable docker && systemctl status docker


注意：
1.Linux8无法直接安装containerd.io
3.启动docker后报错：Your kernel does not support cgroup blkio weight_device
原因是kernel版本过高，建议使用长期维护的4.4版本内核

容器代理设置

目的是让docker容器可以使用HTTP/HTTPS/FTP代理服务器, 17.0.7及以上版本的docker可按下面方法实现代理的自动配置. ⚠️采用VPN代理后, 记得从文件/etc/docker/daemon.json中将中国镜像源注释掉.

  • 步骤
    #此例中,我的VPN代理位于宿主机, 地址是10.19.34.1, 端口是10809.
#在启动容器的用户的家目录中创建.
mkdir /root/.docker/
cat >/root/.docker/config.json <<EOF
{
 "proxies":
 {
   "default":
   {
     "httpProxy": "http://10.19.34.1:10809",
     "httpsProxy": "http://10.19.34.1:10809",
     "noProxy": "*.test.example.com,.example2.com,127.0.0.0/8"   #不经过代理的主机,可使用通配*和CIDR
   }
 }
}

#当这个用户启动容器时, 上面的三个变量将自动被传递到容器.
docker run --name test -d reg.zh.com/springboot/stock:v1
docker exec -it test /bin/sh
/ # env
HTTPS_PROXY=http://10.19.34.1:10809
no_proxy=*.test.example.com,.example2.com,127.0.0.0/8
NO_PROXY=*.test.example.com,.example2.com,127.0.0.0/8
https_proxy=http://10.19.34.1:10809
http_proxy=http://10.19.34.1:10809
HTTP_PROXY=http://10.19.34.1:10809

docker服务代理设置

让docker服务通过代理访问国外镜像仓库. ⚠️我的VPN代理服务运行在宿主机, 地址10.19.34.1:10809. 采用VPN代理后, 记得从文件/etc/docker/daemon.json中将中国镜像源注释掉.

vim /usr/lib/systemd/system/docker.service
#在[Service]下面添加代理配置。
Environment="HTTP_PROXY=http://10.19.34.1:10809"
Environment="HTTPS_PROXY=http://10.19.34.1:10809"
Environment="NO_PROXY=127.0.0.1,localhost

systemctl daemon-reload
systemctl restart docker
docker info #可以看到Http Proxy和Https Proxy

Linux Control Group

  • 为什么

    只有运行环境隔离还不够，因为这些进程还是可以不受限制地使用系统资源，比如网络、磁盘、CPU以及内存 等。因此为了让容器中的进程更加可控，Docker 使用 Linux cgroups 来限制容器中的进程对系统资源的使用。

  • 是什么

    能够为一组进程（进程组）分配系统资源如CPU/内存/存储IO/网络IO等。

    • 单位是进程组
    • 管理员可更具体地控制系统资源的分配、优先顺序、拒绝、管理和监控。
    • 可更好地根据任务和用户分配硬件资源，提高总体效率。
  • 具体功能：
    • Resource limitation: 限制资源使用，比如内存使用上限以及文件系统的缓存限制。
    • Prioritization: 优先级控制，比如：CPU利用和磁盘IO吞吐。（先忽略）
    • Accounting: 一些审计或一些统计，主要目的是为了计费。（先忽略）
    • Control: 挂起进程，恢复执行进程。（先忽略）

在实践中，系统管理员一般会利用CGroup做下面这些事（有点像为某个虚拟机分配资源似的）：

  • 隔离一个进程集合（比如：nginx的所有进程），并限制他们所消费的资源，比如绑定CPU的核。
  • 为这组进程分配其足够使用的内存
  • 为这组进程分配相应的网络带宽和磁盘存储限制
  • 限制访问某些设备（通过设置设备的白名单）

我们看到 /sys/fs/cgroup 目录中有若干个子目录，我们可以认为这些都是受 cgroups 控制的资源以及这些资源的信息。

  • blkio — 这个子系统为块设备设定输入/输出限制，比如物理设备（磁盘，固态硬盘，USB 等等）。
  • cpu — 这个子系统使用调度程序提供对 CPU 的 cgroup 任务访问。
  • cpuacct — 这个子系统自动生成 cgroup 中任务所使用的 CPU 报告。
  • cpuset — 这个子系统为 cgroup 中的任务分配独立 CPU（在多核系统）和内存节点。
  • devices — 这个子系统可允许或者拒绝 cgroup 中的任务访问设备。
  • freezer — 这个子系统挂起或者恢复 cgroup 中的任务。
  • memory — 这个子系统设定 cgroup 中任务使用的内存限制，并自动生成内存资源使用报告。
  • net_cls — 这个子系统使用等级识别符（classid）标记网络数据包，可允许 Linux 流量控制程序（tc）识别从具体 cgroup 中生成的数据包。
  • net_prio — 这个子系统用来设计网络流量的优先级
  • hugetlb — 这个子系统主要针对于HugeTLB系统进行限制，这是一个大页文件系统。

以下演示通过cgroup限制一个进程对CPU、内存、网络IO资源的利用，其他功能暂不涉及。

CPU限制

写一个简单的C程序，它可占用CPU到100%，启动后查一下进程的ID。

cat <EOF >hello.c
int main(void)
{
    int i = 0;
    for(;;) i++;
    return 0;
}
EOF

gcc -o hello hello.c
chmod +x hello.c
./hello.c

[root@k8s-node01 ~]# ps -ef | grep hello
root       615  1674 93 22:04 pts/4    00:00:11 ./hello

在cgroup目录下创建hello目录，然后cgroup会自动在该目录中创建各cgroup子系统对应的目录。

mkdir /sys/fs/cgroup/cpu/hello

编辑cpu.cfs_quota_us，20000表示进程的CPU quota是20000us，默认CPU周期是100000us，也就是说进程只能使用20%的CPU带宽。


[root@k8s-node01 hello]# echo '615' > tasks
[root@k8s-node01 hello]# echo 20000 > cpu.cfs_quota_us

编辑tasks，写入进程ID，这样上面的设置就对这个进程生效了。

[root@k8s-node01 hello]# echo '615' > tasks
