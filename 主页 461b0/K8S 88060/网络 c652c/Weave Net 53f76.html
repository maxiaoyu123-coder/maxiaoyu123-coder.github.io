Kubernetes的网络模型

Kubernetes网络模型设计的一个基础原则是：单Pod单IP模型, 就是Pod里的容器共用一个网络栈。该模型的目标是为了每个Pod分配一个Kubernetes集群私有网络地址段的IP地址。通过该IP，Pod能够跨网络与其他Pod，物理机，容器等进行通信。一个Pod内部的所有容器共享一个网络堆栈（相当于一个网络命名空间，它们的IP地址、网络设备、配置等都是共享的），彼此之间通过localhost通信，就像在一个机器上一样。所以可以将Pod简单看成一个独立的虚拟机. Kubernetes为每个pod分配的IP地址都在一个扁平网络地址空间中, 不需要NAT(网络地址转换)这一点非常重要, 因为NAT将网络地址空间分段的做法, 会引入了额外的复杂性.

网络模型有了, 为了保证基于网络模型设计出来的网络方案具有标准化,可扩展和灵活性, K8S采用了CNI(Container Network Interface)规范, 目前已有的网络方案有Calico, Weave和Flannel.

什么是Weave

Weave fdp工作原理

weave有两种模式, sleeve模式和fast data path(fdp)模式, 现在默认使用fdp模式, 因为它的处理性能更高, 本章主要介绍fdp功能。

什么是Weave fdp呢?

Fast data path takes advantage of the kernel's Open vSwitch datapath (ODP) subsystem, which allows the weave router to tell the kernel how to process packets, rather than processing them itself. ODP operates in terms of "flow rules", which say how to handle packets with certain header field values. ODP reports a "miss" event to the weave router when it encounters a packet that doesn't match a flow rule. This happens, for example, for the first packet between a specific source container and destination container. But the weave router then adds an appropriate flow rule, so that further packets can be handled entirely within the kernel. https://github.com/weaveworks/weave/issues/1713

在K8S节点上启动好weave之后, weave会创建两个网桥和一个虚拟路由器. 分别是Linux 虚拟网桥weave-bridge, Open vSwitch 的虚拟网桥datapath bridge和weave router. 它们实现了节点内Pod之间通信以及跨节点Pod之间的通信. weave router负责收集路由信息, weave-bridge和datapath bridge负责数据转发.

实现fdp的组成部分

分别是weave-bridge，datapath bridge，weave router。

weave-bridge

将容器接入weave网络, K8S节点内Pod之间的通信是由weave-bridge实现. weave bridge上以vethwepl开头的网卡用于连接Pod的Pause容器.vethwepl+容器的前7位字母数字@if数字, k8s集群中pod所管理的pause容器（非hostNetwok模式）在启动之后生成的网卡（一端在容器内部上，一端在主机上）.

连接datapath bridge, weave bridge上的vethwe-bridge接口是连接datapath bridge的.

datapath bridge

非本地容器的目的地址的报文将有datapath bridge处理, 它专门负责处理跨K8S节点的Pod间数据转发. datapath bridge也就是ODP, 它有一份flow规则表, 每条flow项目包含了源mac和目的mac以及本节点和对端节点的地址. packet匹配到某条flow, weave就按照这个flow, 将packet封装成vxlan, 通过UDP连接发送到对端节点的UDP67379端口. datapath bridge自己不生成flow规则表, 当packet匹配不到flow时datapath bridge向weave router查询.

Weave Router

它负责控制面的工作,也就是收集用于处理packet的路由信息, 但是参与packet的处理.

  • 负责发现其他K8S节点上Weave peer, 并与其他peer交换拓扑信息, 维护拓扑表, 通过tcp6783.
  • 负责发现其他节点上的容器的mac地址.
  • 在fast data path模式 Weave Router不负责处理报文, 它是作为datapath bridge的顾问.

地址管理

IPAM

负责管理Weave网络的地址分配.

整个Weave网络的IP地址都来自于一个称为initial range的网段, 这个段由用户定义, 也就是说所有IPAM的initial range都需要设成一样, 否则不能加入Weave网络. 然后IPAM们协商划分出各自的地址范围作为容器的地址池, 避免地址冲突. IPAM没有将initial range拆成更小的子网段, 而是地址范围, 如此各节点上的容器都在同一网段内.

当新的节点加入Weave网络时, 新节点的IPAM会获得IP地址池. 注意下图中的initial allocation是172.16.16.0/24, docker1-4划分出的是地址范围, 它们分配给容器的掩码都是24位, 而不是25位或27位掩码.

weavenet中数据包转发流程

(1) 容器调用内核协议栈封装成一个完整的MAC帧从容器的虚拟接口veth0发送到weave-bridge；

(2) weave-bridge收到数据包后转发到内核中的fastpath网桥，由它进行处理;

(3) datapath module收到数据包后，会查找flow 规则表，若在flow table中找不到匹配的表项，则会产生一个upcall，即向weave router发送一个请求，weave router会下发flow table表项，一旦匹配到相应的flow table表项，数据包即可进行转发；

(4) 在datapath-bridge中，会有一个接口会被配置成vxlan类型的接口，一个vxlan类型的接口会对应一个UDP socket: 数据包从vxlan类型的接口出来后还要经过该UDP socket封装成VxLAN类型的数据包，然后发送到物理网络。例如上图datapath-bridge中有一个vxlan类型的接口vxlan-6784，当数据包要从vxlan-6784接口转发出去时，因为vxlan-6784是vxlan类型的接口，所以该数据包会经过监听在6784端口的UDP socket封装成VxLAN类型的数据包，然后发送出去。

关于Fastpath模式, 你还需要注意:

  • 避免经过NAT设备.
  • 节点上Weave routers使用UDP 6784端口, 保证不被占用.
  • Ensure that WEAVE_MTU fits with the MTU of the intermediate network (see below)
  • Linux内核3.12及之后才支持ODP和VXLAN, 否则将采用sleeve模式.
  • Weave net 1.2及之后, fastpath模式是自动启用的.
  • 当节点上的WeaveNet无法与另一个节点上的WeaveNet建立FastPath连接时, 它会使用更慢的sleeve模式.

sleeve模式

(不作太多研究, 默认使用fdp模式, 仅在条件不允许时才降级到sleeve模式)非fastpath是指不启用fastpath, 也叫sleeve mode或user mode, 通过pcap设备在Linux bridge上截获数据包并由Weave router完成UDP封装,但是性能损失明显,因为Weave router是用户空间程序, 所以整个流程是在用户空间和内核空间之间多次传递,造成上下文频繁切换,效率上比fastpath低.

  1. 源节点Pod向目的节点Pod发出IP报文, 源和目的地址是源节点Pod和目的节点Pod, 报文送至内核.
  1. 内核根据路由表将IP报文送至本节点的WeaveRouter.
  1. WeaveRouter查到目标节点Pod位于目的节点, 然后将IP报文封装在隧道协议头送至内核.
  1. 内核封装到IP报文中, IP报文的目的地址是目的节点.
  1. 目的节点内核收到后解封装IP报文, 当发现隧道协议头时送至本节点的Weave Router程序.
  1. WeaveRouter解封装隧道协议头, 将IP报文送至内核.
  1. 内核解封装IP报文, 然后发送到Pod.
