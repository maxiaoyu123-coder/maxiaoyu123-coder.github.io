什么是kube-proxy

kube-proxy is a key component of any Kubernetes deployment. Its role is to load-balance traffic that is destined for services (via cluster IPs and node ports) to the correct backend pods.Kube-proxy can run in one of three modes, 三种模式userspace/iptables/IPVS。userspace太老了不做介绍。

iptables模式是什么？

iptables是Linux内核的一个特性，一个非常强大的防火墙，可以实现Packet转发控制和过滤。kube-proxy通过制定iptables rule来实现kube-proxy的NAT和负载均衡功能。它的缺点是随着service增加，iptables rules条目变得越来越多，越影响kube-proxy的处理性能，所以kube-proxy采用IPVS来克服这样的问题。

IPVS模式是什么？

IPVS (IP Virtual Server) is built on top of the Netfilter and implements transport-layer load balancing as part of the Linux kernel.它可以为一个服务器集群提供负载均衡，IPVS将TCP/UDP请求重定向到后端真正的服务器上，相比iptables，IPVS拥有更好的性能。

如何开启IPVS

要运行4.X之前的内核版本，在使用IPVS之前，请确认以下内核模块已经开启。

yum install ipset ipvsadm -y

cat > /etc/sysconfig/modules/ipvs.modules <<EOF
#!/bin/bash
modprobe -- ip_vs
modprobe -- ip_vs_rr
modprobe -- ip_vs_wrr
modprobe -- ip_vs_sh
modprobe -- nf_conntrack_ipv4
EOF
chmod 755 /etc/sysconfig/modules/ipvs.modules && bash /etc/sysconfig/modules/ipvs.modules && lsmod | grep -e ip_vs -e nf_conntrack_ipv4

在5.4的内核版本没有nf_conntrack_ipv4模块，但是有nf_conntrack

yum install ipset ipvsadm -y

cat > /etc/sysconfig/modules/ipvs.modules <<EOF
#!/bin/bash
modprobe -- ip_vs
modprobe -- ip_vs_rr
modprobe -- ip_vs_wrr
modprobe -- ip_vs_sh
modprobe -- nf_conntrack
EOF
chmod 755 /etc/sysconfig/modules/ipvs.modules && bash /etc/sysconfig/modules/ipvs.modules && lsmod | grep -e ip_vs -e nf_conntrack_ipv4

Kube proxy开启IPVS模式

kubeproxy启动时加入--proxy-mode=ipvs选项，即可开启IPVS模式。

kubeproxy \
--ipvs-scheduler=wrr #默认是rr，
--ipvs-min-sync-period=5s #最短多久，ipvs规则会被刷新一遍。
--ipvs-sync-period=5s  #最长多久，ipvs规则会被刷新一遍。
--proxy-mode=ipvs #肯定采用ipvs

Service

  • 介绍

    是一组Pod应用的抽象，相当于代理，为Pod提供了持久的VIP

    Service controller与kubeproxy与CoreDNS是联动的

    1. Service负责更新SVC对象在ETCD的状态
    1. kubesystem根据状态，创建删除SVC，更新Endpoints相关的负载均衡规则
    1. CoreDNS根据状态，创建删除SVC以及Endpoints的DNS纪录

    负载均衡Pod controller在添加和删除Pod时，能随时更新

    在负载均衡时，如Service无法将请求转发至某个Pod，那么不会再发送至另一个Pod，即这个请求被丢弃了,故一定要为容器创建readiness probe。

  • 主要类型：NodePort，ClusterIP，LoadBalancer,headless

    NodePort

    除了多出一个地址转换的环节：在物理接口上DNAT+SNAT，DNAT指向SVC的ClusterIP，而SNAT则指向物理接口IP，在创建流程以及流量模型上和ClusterIP一样

    headless

    一种特殊的Service，当你不需要SVC的负载均衡和VIP功能，希望直接访问这些POD时，可以设置为Headless模式

    Kubeproxy不会为其创建SVC的clusterIP和负载均衡规则(Iptables或LVS)，但是Coredns为其创建的A/AAAA纪录使得访问SVC域名时仍然起到轮询的效果

  • 当创建一个clusterIP的SVC，它的创建流程(这中间略过API Server)：
    1. kubectl敲入：kubectl expose deployment hello-world --port=8080 --target-port=8080 -type=CLusterIP ，创建service的请求发送到ETCD
    1. ETCD将请求保存起来
    1. service controller醒了，查看ETCD中的数据，发现有一个创建service的请求，为其创建Endpoints对象(注册Pods地址和端口信息)，保存在ETCD
    1. 节点上的kubeproxy醒了，它看到ETCD中关于Endpoints对象的数据，它做两件事情1：创建VIP；2：创建负载均衡规则(LVS或者iptables) ⚠️注意：headless没有这一步
    1. kubeproxy创建物理接口与SVC ClusterIP间的DNAT&SNAT的转换 ⚠️注意：NodePort才有这一步
    1. coreDNS发现了新的Service和它的Endpoints，为他们创建A/AAAA纪录
  • 集群内部，某个Pod访问另一个SVC的流量模型

    集群中一个Pod作为客户端访问另一个Serivce

  • 集群外部，某个客户端访问SVC的的流量模型，这个SVC通过NodePort暴露

CoreDNS

介绍

轻量DNS服务，提供服务发现机制

提供这些服务：

监听Service和Endpoint的事件，当Service创建时为其以及Endpoints创建DNS纪录；当Service删除时删除以及Endpoints其DNS纪录; 当Endpoints删创时删创对应的DNS纪录。

当Service创建，对应的A纪录/AAAA纪录，格式<svcname>.<ns>.svc<clusterdomain>，如helloworld.default.svc.cluster.local

每个Pod在/etc/resolv.conf中nameserver的地址是coreDNS的clusterIP; 另外还有search，当输入缩写域名，它能够在解析时自动补全，不需要输入FQDN，在同一namespace时域名用<svcname>即可，如不同namespace则域名用<svcname>.<ns>

在节点上使用nslookup解析

#在节点上解析时因为resolv.conf没有配置search，所以要写FQDN
nslookup <SVC>.<NS>.svc.cluster.local <CoreDNS的CLusterIP>

#比如解析
nslookup elasticsearch.elastic-kibana.svc.cluster.local 10.96.0.10
